# Agent HQ - 開發流程與原則

## 1. 開發流程設計

### 階段流程

```
需求確認 → 規格制定(SPEC) → 架構設計 → 計劃制定(PLAN) → 任務拆分(TASK) → 開發實作 → 測試驗收 → 發布部署
```

| 階段 | 產出 | 負責 |
|------|------|------|
| 需求確認 | 需求描述 | Master → JARVIS |
| 規格制定 | SPEC.md | JARVIS |
| 架構設計 | 架構圖、模組設計 | Claude Code |
| 計劃制定 | PLAN.md | Claude Code |
| 任務拆分 | TASK.md | Claude Code |
| 開發實作 | 原始碼 | Claude Code |
| 測試驗收 | 測試報告 | Claude Code |
| 發布部署 | 可運行專案 | Claude Code |

---

## 2. 角色分工

### 🤖 JARVIS（專案監督者）

**職責**：
- 理解 Master 需求並翻譯
- 呼叫 Claude Code 進行開發
- 監控開發進度
- **回報狀況給 Master**
- **不直接撰寫程式碼**

**原則**：
- 只負責協調與溝通
- 不親自寫程式碼
- 即時回報進度

---

### 💻 Claude Code（實際開發者）

**職責**：
- 撰寫程式碼
- 建構專案
- 修復問題
- 執行測試

**使用方式**：
```bash
# 單一任務
claude -p "<任務描述>"

# Agent Team（多個 Agent 並行）
claude --agents '{
  "agent-name": {
    "description": "...",
    "prompt": "..."
  }
}' -p "<任務描述>"
```

**權限設定**：
- 允許特定工具：`--allowed-tools "Bash(npm:*) Edit Write Read"`
- 需要時詢問 Master 確認

---

### 👤 Master（決策者）

**職責**：
- 提出需求
- 審閱規格與計畫
- 確認驗收
- 提供反饋

---

## 3. 回報機制

### 原則：持續回報，避免 timeout

**之前問題**：
- Claude Code 執行時間過長導致 timeout
- Master 不知道進度

**解決方案**：
1. **先回覆進度**：每次 Claude Code 啟動後，先回覆已開始
2. **定時檢查**：定時查看執行狀態
3. **持續回報**：有進展就回報給 Master
4. **完成確認**：最後確認是否成功

## 5. 設計原則

### 核心原則

1. **基於 Claude Code 介面**
   - 不額外疊加資訊
   - 忠實呈現 Claude Code 的狀態與輸出
   - 所有功能設計都應基於 Claude Code 本身提供的資訊

2. **MVP 優先**
   - 先做出最小可行性產品
   - 後續再優化

3. **動態 Agent 支援**
   - 系統能處理任意數量的 Agent
   - 視覺風格自動對應

## 6. 這次開發的優化

### 相比之前的改進

| 項目 | 之前 | 現在 |
|------|------|------|
| 開發者 | JARVIS 直接寫 | Claude Code 執行 |
| 權限 | 每次詢問 | 預設允許特定工具 |
| 回報 | 完成後才回報 | **持續回報** |
| timeout | 常timeout中斷 | **先回報再持續確認** |
| 測試 | 無 | 76 個測試 |
| 文件 | 分散 | 統一在 `.agent/doc/` |

### 新增原則

1. **持續回報機制**
   - 不再等待 Claude Code 完成
   - 先回覆進度，定時檢查
   - 完成後確認結果

2. **權限預設**
   - 使用 `--allowed-tools` 預設允許
   - 避免每次詢問

3. **文件統一**
   - 所有開發文件在 `.agent/doc/`
   - 方便追蹤與 review
